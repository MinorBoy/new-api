# Seedance 参数能力路由设计

## 背景

管理员需要把多家上游渠道的 Seedance 系列模型统一到本站模型 ID。客户端只提交本站公开的官方模型 ID；new-api 根据用户分组、本站模型和请求参数选择兼容渠道，并把模型名改写为该渠道实际使用的上游模型 ID。

现有模型映射已经能表达一对一别名。例如两个渠道都可以公开 `doubao-seedance-2-0-260128`，再分别映射到各自的上游模型。但是当前分发阶段只读取 `model` 和 `group`，在完整视频请求解析之前就完成渠道选择；`abilities` 和候选缓存也只包含分组、模型、渠道、优先级与权重。因此它不能依据分辨率、时长、比例、参考素材或真人能力选择渠道，也不能在同一渠道内为一个本站模型选择多个上游模型。

现有渠道选择链路已经提供分组、优先级、权重、自动分组、亲和、重试和自动禁用。新能力应复用这条链路，而不是复制一套 Seedance 专用分发器。

## 目标

1. 客户端只看到并提交三个官方本站模型 ID：
   - `doubao-seedance-2-0-260128`
   - `doubao-seedance-2-0-fast-260128`
   - `doubao-seedance-2-0-mini-260615`
2. 按具体分组、本站模型和规范化请求事实过滤不兼容的上游目标。
3. 一个渠道可以为同一本站模型配置多个上游路由目标。
4. 渠道内部按目标优先级确定上游模型；渠道之间继续使用现有优先级和权重。
5. 保留现有亲和、重试、自动禁用、计费和任务生命周期。
6. 模型广场、模型列表、任务响应和用户日志始终使用本站模型 ID，不暴露上游模型 ID。
7. 配置、迁移和查询同时支持 SQLite、MySQL 5.7.8+ 和 PostgreSQL 9.6+。

## 非目标

- 不分析提示词、图片或视频来推断是否包含真人。
- 不下载远程素材来检查内容、尺寸或时长。
- 不在第一阶段实现最低上游成本路由。
- 不把上游成本配置与用户计费规则合并。
- 不增加自由表达式路由语言。
- 不为 Seedance 创建独立的重试、亲和、自动禁用或计费链路。
- 不自动把现有 `model_mapping` 转换为能力规则，因为旧配置没有足够的能力元数据。
- 不在第一阶段把 `generate_audio`、`service_tier`、`draft` 或 `tools` 加入目标匹配；这些字段继续由现有通用校验和渠道适配器校验。

## 方案选择

### 采用：现有路由链路加独立能力策略层

能力层在现有候选渠道与优先级/权重选择之间工作。它负责提取请求事实、匹配渠道内目标，并向现有选择器提供本次请求允许的渠道集合。选择器仍然负责组内优先级和权重。

该方案保留当前稳定的渠道生命周期，同时让参数匹配成为可复用于其他视频模型的通用能力。

### 不采用：直接扩充 `abilities`

`abilities` 是由渠道配置派生的轻量候选索引。把分辨率、时长区间、比例、素材上限和多个上游模型写进该表，会让一条能力记录承担候选索引、条件规则和模型映射三种职责，也无法自然表达同一渠道的多个目标。

### 不采用：Seedance 专用路由器

独立路由器需要复制自动分组、渠道状态、亲和、重试、优先级、权重、自动禁用、日志和计费衔接。两套路由器会产生不同的错误和重试行为，后续维护成本高于扩展现有选择入口。

## 外部 API 契约

Seedance 官方创建请求继续使用现有入口：

```text
POST /api/v3/contents/generations/tasks
```

客户端提交本站模型 ID 和官方请求字段。为不能从官方字段可靠推断的真人要求，增加一个本站扩展对象：

```json
{
  "model": "doubao-seedance-2-0-260128",
  "content": [
    { "type": "text", "text": "A person walking through a station" },
    {
      "type": "image_url",
      "role": "reference_image",
      "image_url": { "url": "https://example.com/reference.jpg" }
    }
  ],
  "resolution": "1080p",
  "duration": 10,
  "ratio": "16:9",
  "routing": {
    "require_real_person": true
  }
}
```

`routing.require_real_person` 是可选布尔值。只有显式 `true` 才要求目标声明支持真人；缺省或 `false` 不增加限制。`routing` 在选择完成后移除，绝不转发给上游。

能力模式第一阶段只接受正整数时长。官方智能时长 `duration=-1` 返回 `no_compatible_route`，因为当前收集的上游目标都只声明固定值或正整数区间；未来只有在能力结构增加明确的智能时长字段后才能启用。

第一阶段只为 Seedance 官方 JSON 创建请求启用完整能力提取。现有 OpenAI 视频、Kling、Jimeng 和其他任务入口继续使用 `legacy` 模式；后续可以通过新增事实提取器接入同一个能力层，不改变数据模型和选择算法。

## 总体架构

```text
Seedance 官方请求
  -> TokenAuth 确定用户与令牌分组
  -> 提取并校验 RoutingFacts
  -> 按具体 group + canonical model 读取 RoutingPolicy
  -> 读取现有 ability 候选渠道
  -> 在每个渠道内过滤 RouteTarget
  -> 渠道内选择最高 target_priority 的唯一目标
  -> 现有渠道优先级分层 + 同层权重选择
  -> 把 target_id 和 upstream_model 写入 RelayInfo/context
  -> 适配器校验、计费、提交和任务持久化
```

能力模式不是第二套路由器。它是 `Distribute` 和现有渠道选择服务共同使用的一个可选过滤步骤。

有效路由模式如下：

- `legacy`：该具体分组和本站模型没有启用的能力策略，完全执行当前行为。
- `capability`：存在启用的能力策略，必须命中一个启用目标。没有目标的渠道不能回退到普通 `model_mapping`。

禁用能力策略即可回退到 `legacy`，用于渐进上线和紧急回滚。

## 领域模型

### RoutingPolicy

一条策略对应一个具体分组和一个本站模型。

| 字段 | 含义 |
|---|---|
| `id` | 策略主键 |
| `group_name` | 具体用户分组，避免使用数据库保留字 `group` |
| `model` | 本站公开模型 ID |
| `enabled` | 是否启用能力模式 |
| `default_resolution` | 请求缺省时使用的本站分辨率默认值 |
| `default_duration` | 请求缺省时使用的本站整数秒默认值 |
| `default_ratio` | 请求缺省时使用的本站比例默认值 |
| `created_at` / `updated_at` | 审计时间 |

`(group_name, model)` 必须唯一。策略启用时，三个默认值必须合法，并且至少有一个启用目标能匹配完整默认请求。

自动分组继续按现有顺序展开为具体分组。每个具体分组独立读取策略和目标，不创建名为 `auto` 的共享策略。

### RouteTarget

路由目标表示某个渠道在该策略下可选择的一个上游模型和能力范围。

| 字段 | 含义 |
|---|---|
| `id` | 目标主键 |
| `policy_id` | 所属能力策略 |
| `channel_id` | 现有渠道 ID |
| `name` | 管理员可读名称 |
| `upstream_model` | 最终发送给该渠道的上游模型 ID |
| `target_priority` | 同一渠道内部的目标优先级，数值越大越优先 |
| `constraints` | 使用 TEXT 保存的结构化 JSON 能力约束 |
| `enabled` | 是否参与匹配 |
| `created_at` / `updated_at` | 审计时间 |

一个策略可以关联多个渠道，一个渠道在同一策略下也可以有多个目标。渠道删除时显式删除其目标；策略删除时显式删除所属目标，不依赖特定数据库的级联行为。

`constraints` 通过 `common.Marshal` 和 `common.Unmarshal` 编解码，不使用数据库 JSON 查询或数据库专用 JSON 类型。服务启动和配置更新时把它解析为类型化内存结构。

## 能力约束

目标能力使用固定结构，不接受任意表达式：

```json
{
  "output_resolutions": ["1080p"],
  "durations": { "min": 4, "max": 15 },
  "aspect_ratios": ["16:9", "9:16"],
  "reference_limits": {
    "images": 9,
    "videos": 3,
    "audios": 3
  },
  "supports_real_person": true
}
```

约束语义：

- `output_resolutions` 是用户请求匹配字段，使用 `480p`、`720p`、`1080p` 或 `4k`。
- 上游内部如何得到输出分辨率不属于路由能力。例如渠道将内部 720p 结果超分为 1080p 时，该目标仍只配置 `1080p` 输出能力；内部实现可以写在目标名称中，但不作为路由字段或上游参数。
- `durations` 使用离散 `values` 或闭区间 `min/max`，两种形式互斥。Seedance 路由只接受整数秒。
- `aspect_ratios` 缺省或空数组表示不限制，否则必须精确命中规范比例。
- `reference_limits` 三个字段均为必填的有界非负整数。`0` 表示不支持该素材类型。
- `supports_real_person` 是可空布尔值。`null` 表示能力未知；真人要求为 `true` 时只有明确的 `true` 可以命中。

保存时还必须满足：

- `output_resolutions` 非空且只包含 `480p`、`720p`、`1080p` 或 `4k`。
- 离散时长和区间端点都在 `1..relaycommon.MaxTaskDurationSeconds` 内。
- 比例只包含当前 Seedance 官方集合：`16:9`、`4:3`、`1:1`、`3:4`、`9:16`、`21:9`、`adaptive`。
- 图片、视频和音频上限分别在 `0..9`、`0..3` 和 `0..3` 内，与现有 Seedance 请求全局边界一致。
- `adaptive` 与视频或音频输入的冲突继续由官方请求语义校验拒绝，不由独立目标规则表达。

采集表中的素材缩写按以下方式展开：

| 原值 | 图片上限 | 视频上限 | 音频上限 |
|---|---:|---:|---:|
| `933` | 9 | 3 | 3 |
| `431` | 4 | 3 | 1 |
| `9` | 9 | 0 | 0 |

上游计价单位和单价不属于能力约束。第一阶段不根据每秒、每次或每百万 token 成本选择目标。

## RoutingFacts

路由前生成不可变的规范化请求事实：

```text
GroupName
CanonicalModel
OutputResolution
DurationSeconds
AspectRatio
ReferenceImageCount
ReferenceVideoCount
ReferenceAudioCount
RequireRealPerson
```

值的优先级是：显式请求值高于策略默认值。目标自己的默认值不能参与选择，否则同一客户端请求会因随机渠道不同而产生不同输出。

事实提取只读取 JSON 字段和 `content[].type/role`：

- 不分析文本内容。
- 不识别人脸。
- 不下载媒体 URL。
- 不读取远程视频时长。
- 图片、视频和音频数量按请求中明确提交的条目计算。

提取器复用 `common.GetBodyStorage`，保证请求体只缓存一次且后续适配器仍可读取。规范化结果存入 Gin context，后续选择、适配器校验、计费快照和日志复用同一份事实。

## 匹配和选择算法

对每个具体分组执行以下步骤：

1. 按 `(group_name, canonical_model)` 查找启用策略。
2. 没有启用策略时执行现有 `legacy` 选择。
3. 有策略时补全默认参数并校验 `RoutingFacts`。
4. 对策略中的启用目标执行所有能力约束匹配。
5. 按 `channel_id` 分组目标；每个渠道只保留最高 `target_priority` 的兼容目标。
6. 与现有 `abilities` 候选渠道求交集，排除禁用、不可用或不支持该本站模型的渠道。
7. 把剩余渠道交给现有优先级分层和同层权重随机算法。
8. 将选中渠道对应的唯一目标写入请求上下文。

配置保存时检查同一策略、同一渠道、相同 `target_priority` 的目标是否可能匹配同一个请求。只要分辨率、时长、比例、素材上限和真人要求的交集非空，就视为重叠并拒绝保存。运行时仍以目标 ID 作稳定排序用于防御，但正常配置不会依赖该兜底。

渠道优先级和权重仍来自现有渠道/能力索引；`target_priority` 只解决同一渠道内部的上游模型选择，不参与跨渠道负载均衡。

## 模型映射和计费边界

能力模式下，`RouteTarget.upstream_model` 是最终上游模型名，优先于该渠道的普通 `model_mapping`，且不能再次被映射。

路由完成后必须保持：

```text
OriginModelName   = 客户端提交的本站模型 ID
UpstreamModelName = RouteTarget.upstream_model
```

用户定价、模型倍率、分组倍率、预扣费和结算继续使用 `OriginModelName`。任务私有属性和管理员日志记录 `UpstreamModelName`、策略 ID 与目标 ID。用户响应、模型广场和非管理员日志不展示上游模型。

上游成本和用户计费保持独立。若未来增加最低成本策略，需要先把每秒、每次和每百万 token 单价转换为针对当前请求的统一成本估算；该能力不在本设计范围内。

## 亲和、重试和渠道状态

- 亲和缓存命中的渠道必须针对当前 `RoutingFacts` 重新计算兼容目标；不兼容时忽略该亲和结果。
- 每次重试重新进入同一个能力过滤和渠道选择入口。
- 首期重试失败后排除整个渠道，不在同一渠道内静默切换第二目标，保持现有失败域语义。
- 自动禁用、渠道测试状态、多 Key 行为和最大重试次数继续使用现有逻辑。
- 锁定渠道也必须匹配能力目标；锁定渠道不兼容时返回本地错误，不绕过策略。

## 错误语义

错误按以下顺序分类：

| 场景 | HTTP 状态 | 稳定错误码 |
|---|---:|---|
| 请求字段类型、值或素材数量本身非法 | 400 | 对应的 `InvalidParameter.*` |
| 请求合法，但没有任何启用目标支持该参数组合 | 400 | `no_compatible_route` |
| 存在兼容目标，但对应渠道全部禁用、不可用或重试耗尽 | 503 | `compatible_channel_unavailable` |
| 路由配置损坏或缓存无法加载 | 500 | `routing_policy_error` |

任何场景都不降级到不兼容目标。普通错误响应不包含渠道名、渠道 ID、目标 ID 或上游模型。

管理员日志记录规范化事实、策略 ID、最终目标 ID，以及被过滤目标的聚合原因，例如 `resolution`、`duration`、`aspect_ratio`、`reference_images`、`reference_videos`、`reference_audios` 或 `real_person`。日志不记录提示词和素材 URL。

## 缓存与一致性

能力缓存按 `(group_name, canonical_model)` 保存完整不可变策略快照，并在快照内按渠道索引目标。请求热路径不访问数据库，也不解析 JSON。

创建、更新、启用、禁用或删除策略/目标时：

1. 在数据库事务内校验完整策略和所有目标。
2. 提交事务。
3. 重新加载该 `(group_name, canonical_model)` 快照并原子替换缓存。
4. 失效相关的现有候选渠道缓存。

渠道模型、分组、状态或删除操作继续更新 `abilities`，并同时使关联策略快照失效。缓存重建失败时保留上一个完整快照并记录错误；不得发布部分策略。

## 管理 API 与界面

后台新增独立的“模型路由策略”管理页，而不是把复杂目标编辑器嵌入渠道表单。页面提供：

- 按分组和本站模型筛选的策略列表。
- 新建、编辑、启用、禁用和删除策略。
- 在策略内新增、复制、编辑、排序、启用、禁用和删除路由目标。
- 渠道选择器只允许选择当前分组和本站模型的现有候选渠道。
- 结构化控件编辑分辨率、允许生成时长、比例、素材上限和真人支持。
- 分组从现有管理员分组接口加载；抽屉中的下拉列表统一使用项目主题化 Select/Combobox。
- 选中和启用状态使用语义 `primary` 主题色，深色主题呈现项目统一的偏蓝色视觉效果。
- 保存前展示字段错误和目标重叠错误。
- 渠道页面只显示关联路由目标数量并提供跳转入口。

读取接口复用渠道查看权限，写入接口复用渠道写权限。上游模型 ID 仅管理员可见；API Key 不进入策略响应。

所有用户可见文本使用项目现有 i18n，覆盖 `en`、`zh`、`fr`、`ja`、`ru` 和 `vi`。

## 数据库与迁移

新增策略表和目标表，通过 GORM `AutoMigrate` 创建。约束 JSON 使用 `TEXT`。查询、唯一索引和删除使用 GORM API，避免方言专用 SQL。

升级后默认没有启用策略，因此所有现有渠道继续使用 `legacy` 行为。管理员完成目标配置并启用某个具体分组/模型策略后，该组合才进入 `capability` 模式。普通 `model_mapping` 可以保留用于快速回滚，但能力模式下不参与最终上游模型选择。

第一批策略建议只覆盖 `分组A` 和三个 Seedance 2.0 本站模型。A1、A1_copy 以及后续渠道逐个录入目标并验证后再启用，避免一次迁移全部渠道。

## 验证与验收

### 后端测试

- 类型化约束解析、规范化和保存校验。
- 离散时长、闭区间、任意比例、输出分辨率和素材上限匹配。
- 真人能力的 `true`、`false` 和未知语义。
- 同渠道同优先级目标重叠检测。
- 请求默认值补全和显式值优先级。
- `933`、`431` 和 `9` 三种素材限制组合。
- 能力目标过滤后仍遵守渠道优先级和权重。
- 亲和命中不兼容时失效，重试排除失败渠道。
- `400 no_compatible_route` 与 `503 compatible_channel_unavailable` 的区分。
- `OriginModelName` 保持本站模型，`UpstreamModelName` 使用目标模型。
- 预扣费、结算、任务属性和用户响应不因上游模型改写而改变本站计费身份。
- 缓存原子替换、配置失效和渠道删除清理。
- SQLite 测试覆盖迁移和 CRUD；现有 CI 继续覆盖 MySQL/PostgreSQL 兼容性。

### 前端测试

- 策略和目标表单 schema 校验。
- 时长离散值/区间互斥。
- 分组数据加载、主题化下拉、主色选中状态和三个素材上限控件。
- 保存错误映射、权限状态和渠道过滤。
- 所有新增翻译键同步。

### 端到端验收
